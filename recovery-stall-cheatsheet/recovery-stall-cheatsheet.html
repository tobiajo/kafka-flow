Transactional recovery read — stall cheat sheet

KafkaPartitionPersistence · read_committed drain to the captured high‑watermark · poll 10 ms · deadline 3 min · legit wait ~70 s (txn.timeout 60 s + abort scan 10 s) · worked example HW = 1000

Control flow one loop, two kinds of turn, two exits

SETUP (once, before the loop)
  L1  highWatermark    = endOffsets(read_uncommitted)      · the high watermark
  L2  lastStableOffset = endOffsets(read_committed)        · the LSO (reader's own end)
  L3  if lastStableOffset < highWatermark   →  WARN "recovery waits for open transaction(s)"
  L4  targetOffset     = highWatermark                · the target: captured once, fixed for the read (never the LSO)

LOOP  (tailRecM · one turn ≈ 10 ms)
  L5  offset = position()
      └─ if offset ≥ targetOffset ───────────────────►  EXIT ✓  read complete
  L6  advanced?  (offset ≠ last.position)
      ├─ YES → ADVANCING turn · reset clock (progressAt = now)
      └─ NO  → WAITING turn · L7 if now − progressAt ≥ 3 min ──►  EXIT ✗  L8 RecoveryReadStalledError
                                                    L9 logEnd < target ? truncation : outlived
                              else · every 5 s → INFO "no progress …"
  ↻   both turns end:  poll(10.millis) ; fold batch ──────────►  back to L5

  • Every scenario runs the same loop — each turn is either an advancing turn (L6: poll returned a batch, position climbs) or a waiting turn (L7: poll came back empty, position frozen, clock ticks). A is all advancing turns; B/C/D add a run of waiting turns.
  • B ≡ C until 70 s — identical waiting-turn runs; the only difference is whether a poll returns records before the clock reaches L8. No separate "hang" detector — the deadline is it, discovered late on an ordinary turn, never a scheduled wake-up.
  • C vs D differ only at L3 (warn fires / silent) and L9 (logEnd vs target). Reader is group-less, so max.poll.interval.ms eviction can't unwind the loop — only ✓ L5 (records) or ✗ L8 (deadline) exits it.

Caveats & operations truncation is a Kafka durability boundary, not a bug

  • Two stalls, two shapes — B/C stall at the LSO: committed records sit above, hidden by an open transaction (they appear once it resolves). D stalls at the true log end: those records are gone (truncated).
  • Detection is in-flight only — the deadline catches truncation that lands during a read. A truncation between reads is silent: the next recovery adopts the regressed end as its target and completes, possibly resuming stale state from an ahead offset — the #732 shape.
  • Reduce the loss risk — the deadline only flags lost records; it can't recover them. Keep the snapshot topic durable: unclean.leader.election.enable=false, min.insync.replicas ≥ 2, RF ≥ 3 (the transactional producer already forces acks=all).
  • On RecoveryReadStalledError — truncation → offset-reset / restore (don't blind-restart: it recovers silently to the regressed end); outlived → wait, or abort a hanging txn with kafka-transactions.sh.
